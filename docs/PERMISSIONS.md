# Роли и разрешения

## 1. Принцип

В интерфейсе используются три клиентские роли, но сервер проверяет конкретные права и область действия.

Роль отвечает на вопрос «что можно делать», область — «с какими объектами и людьми».

## 2. Роли

### Подрядчик

Полный доступ внутри своей организации.

### Бригадир

Операционный доступ к назначенным объектам, командам или сотрудникам. Бригадир необязателен и не разрывает прямую связь подрядчика с работником.

### Работник

Доступ к собственным сменам, задачам, сообщениям, результатам и табелю.

### Администратор платформы

Служебный доступ для поддержки. Не участвует в строительных процессах и не должен использовать клиентские права для обычной работы.

## 3. Области доступа

Поддерживаются:

- вся организация;
- конкретный объект;
- несколько объектов;
- конкретная команда;
- выбранные сотрудники;
- собственные данные.

Бригадиру обычно назначается объектная область. Работнику — собственные данные и общие задачи, где он является участником.

## 4. Совмещение ролей

Один аккаунт может:

- быть бригадиром на объекте А;
- быть исполнителем на объекте Б;
- выполнять собственные задачи;
- управлять назначенной командой.

Интерфейс показывает объединение доступных функций без создания второго аккаунта.

## 5. Матрица разрешений

Обозначения:

- **Да** — разрешено в пределах области.
- **Свои** — только собственные данные.

- **Нет** — запрещено.

## 6. Прямое взаимодействие подрядчика и рабочего

Подрядчик всегда может:

- назначить задачу напрямую;
- изменить исполнителя;
- написать рабочему;
- ответить на проблему;
- проверить результат;
- скорректировать смену;
- увидеть историю.

Наличие бригадира не перенаправляет и не блокирует эти действия.

Работник может напрямую обратиться к подрядчику в рамках своей организации. При желании организация сможет позднее настроить маршрутизацию, но MVP не запрещает прямую связь.

## 7. Права бригадира

По умолчанию бригадир может в своей области:

- видеть присутствие;
- управлять задачами;
- назначать сотрудников из доступного состава;
- проверять результаты;
- решать проблемы;
- вносить корректировки с причиной;
- видеть отчёты объекта.

Не может:

- управлять владельцами организации;
- закрывать месяц;
- менять тарифы;
- получать доступ к другим объектам;
- удалять историю;
- видеть служебные настройки платформы.

## 8. Данные участников общей задачи

Работник видит участников общей задачи в объёме, необходимом для работы:

- имя и инициалы;
- аватар;
- роль в задаче;
- рабочую специализацию;
- статус ответственности.

Телефон и личные данные других сотрудников по умолчанию не показываются.

## 9. Увольнение или снятие с объекта

После прекращения доступа пользователь:

- не получает новые задачи и сообщения объекта;
- не может открыть новые приватные файлы;
- сохраняется автором старых событий;
- остаётся в закрытых табелях и отчётах;
- может видеть собственную разрешённую историю согласно политике организации.

## 10. Проверка разрешений

Frontend использует права для UX, но backend выполняет окончательную проверку.

Каждый запрос проверяет:

1. сессию;
2. организацию;
3. членство;
4. объектную область;
5. действие;
6. состояние сущности;
7. дополнительные ограничения, например статус закрытого периода.

## 11. Нельзя расширять права косвенно

- Участие в задаче не даёт доступ ко всему объекту.
- Доступ к объекту не даёт доступ ко всей организации.
- Ответственный за отчёт не становится бригадиром.
- Автор проблемы не получает право менять задачу.
- Создатель комментария не получает право менять официальный результат.
- Ссылка на файл не отменяет проверку доступа.

## 12. Будущие роли

Архитектура должна позволить позднее добавить:

- бухгалтера;
- инженера;
- специалиста по охране труда;
- заказчика-наблюдателя;
- менеджера организации.

Они не входят в MVP и не должны появляться в пользовательском интерфейсе раньше необходимости.
