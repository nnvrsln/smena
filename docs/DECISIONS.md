# СМЕНА — журнал продуктовых и технических решений

## Правила журнала

- Решение не удаляется после изменения — оно получает статус «Заменено» и ссылку на новое.
- Утверждённое решение меняется только после оценки влияния на спецификацию, UX, данные и приёмку.
- Краткий список ниже фиксирует именно принятые решения, а не весь backlog идей.

## Принятые решения

### DEC-001 — Три клиентские роли

- **Статус:** утверждено.
- **Решение:** подрядчик, бригадир, работник.
- **Причина:** роли соответствуют реальному управлению объектом и понятны пользователям.
- **Следствие:** служебный администратор платформы существует отдельно и не является четвёртым кабинетом клиента.

### DEC-002 — Бригадир необязателен

- **Статус:** утверждено.
- **Решение:** рабочий не обязан быть прикреплён к бригадиру.
- **Причина:** небольшими командами подрядчик управляет напрямую.
- **Следствие:** подрядчик всегда может назначить задачу, написать, решить проблему и проверить результат любого работника.

### DEC-003 — Команды необязательны

- **Статус:** утверждено.
- **Решение:** задача назначается одному человеку, выборке людей или сохранённой команде.
- **Причина:** реальный состав работ часто меняется.

### DEC-004 — Одна универсальная форма задачи

- **Статус:** утверждено.
- **Решение:** пользователь не выбирает «простая/объёмная/составная» в начале.
- **Причина:** тип появляется автоматически после добавления объёма или этапов.
- **Отклонено:** обязательный мастер выбора технического типа.

### DEC-005 — Один уровень этапов

- **Статус:** утверждено.
- **Решение:** задача → этапы, без подэтапов бесконечной глубины.
- **Причина:** глубокие деревья усложняют создание и выполнение.

### DEC-006 — Нет обязательного почасового расписания задач

- **Статус:** утверждено.
- **Решение:** подрядчик не обязан указывать, с какого по какое время человек выполняет конкретную работу.
- **Причина:** стройка постоянно меняется; плановые интервалы создают ложную точность и лишнюю работу.
- **Отклонено:** автоматический рабочий контекст на основе заранее заданных часовых блоков.

### DEC-007 — Нет обязательных таймеров задач

- **Статус:** утверждено.
- **Решение:** точное время считается по смене; участие и результат фиксируются по задачам.
- **Причина:** рабочий не должен постоянно открывать приложение.

### DEC-008 — Результат можно вносить весь день

- **Статус:** утверждено.
- **Решение:** простое выполнение, объём, этап, фото, комментарий и проблема доступны в любое время.
- **Следствие:** экран завершения смены только собирает итог и напоминает о пропусках.

### DEC-009 — Единственный исполнитель отвечает автоматически

- **Статус:** утверждено.
- **Решение:** если задача назначена одному человеку, он автоматически является ответственным за отчёт.

### DEC-010 — Один ответственный за общий командный результат

- **Статус:** утверждено.
- **Решение:** общий числовой итог сдаёт один ответственный.
- **Fallback:** если его не выбрали, участник может «взять ответственность», а руководитель назначить или заменить.
- **Причина:** исключение двойного учёта одного и того же объёма.

### DEC-011 — Командный объём не делится автоматически

- **Статус:** утверждено.
- **Решение:** команда получает общий результат, а сотрудники — факт участия.
- **Причина:** автоматическое деление создаёт недостоверную личную производительность.

### DEC-012 — Разделение общего, дневного и подтверждённого объёма

- **Статус:** утверждено.
- **Решение:** общий объём работы, план дня, заявленный факт и подтверждённый факт — разные значения.

### DEC-013 — Остаток переносится без копирования задачи

- **Статус:** утверждено.
- **Решение:** многодневная задача сохраняет единую историю и получает новый дневной план.

### DEC-014 — Простые задачи не требуют обязательной приёмки

- **Статус:** утверждено.
- **Решение:** простая задача закрывается исполнителем, если руководитель не включил проверку.
- **Причина:** подрядчик не должен подтверждать десятки бытовых поручений.

### DEC-015 — Табель строится из событий

- **Статус:** утверждено.
- **Решение:** начало, обед, переход, завершение и корректировки сохраняются как события.
- **Причина:** доказуемость и возможность восстановить историю.

### DEC-016 — Одна открытая смена

- **Статус:** утверждено.
- **Решение:** один человек не может иметь две параллельные смены.
- **Следствие:** переход между объектами создаёт сегменты одной смены.

### DEC-017 — Два режима обеда

- **Статус:** утверждено.
- **Решение:** фиксированный обед вычитается по политике объекта; свободный отмечается вручную.

### DEC-018 — Корректировки не уничтожают исходные данные

- **Статус:** утверждено.
- **Решение:** изменение времени или результата содержит автора, причину, старое и новое значение.

### DEC-019 — Управление табелем по исключениям

- **Статус:** утверждено.
- **Решение:** нормальные дни подтверждаются группой, спорные попадают в отдельную очередь.

### DEC-020 — Offline-first

- **Статус:** утверждено.
- **Решение:** критичное действие сначала сохраняется локально, затем отправляется на сервер.
- **Следствие:** используются UUID операций, очередь и идемпотентность.

### DEC-021 — Фоновая синхронизация не является единственной гарантией

- **Статус:** утверждено.
- **Решение:** очередь также отправляется при открытии, фокусе, появлении сети и вручную.
- **Причина:** возможности закрытой PWA различаются по браузерам.

### DEC-022 — Три пользовательских статуса синхронизации

- **Статус:** утверждено.
- **Решение:** «Отправлено», «Сохранено на телефоне», «Требует проверки».
- **Отклонено:** показ технических статусов pending/retry/conflict.

### DEC-023 — Push только для значимых событий

- **Статус:** утверждено.
- **Решение:** все события остаются в центре активности, но push фильтруется по важности и ответственности.

### DEC-024 — Комментарий не изменяет задачу

- **Статус:** утверждено.
- **Решение:** официальное изменение плана, срока или объёма выполняется отдельным действием.

### DEC-025 — Контекстные коммуникации

- **Статус:** утверждено.
- **Решение:** задача, проблема, прямой диалог и объявление объекта — отдельные контексты.
- **Отклонено:** общий неструктурированный групповой чат в MVP.

### DEC-026 — Одна кнопка добавления фото

- **Статус:** утверждено.
- **Решение:** нет пользовательского разделения «снято в приложении / выбрано из галереи».

### DEC-027 — Регистрация только по инвайту

- **Статус:** утверждено.
- **Решение:** подрядчик создаёт одноразовую ссылку с ролью и объектами и передаёт её через мессенджер.
- **Отклонено:** SMS-регистрация в MVP.

### DEC-028 — Телефон и пароль

- **Статус:** утверждено.
- **Решение:** телефон — логин, телефон пока не подтверждается, пароль создаёт пользователь.

### DEC-029 — Восстановление через подрядчика

- **Статус:** утверждено.
- **Решение:** запрос, офлайн-проверка личности и одноразовая reset-ссылка.
- **Условие:** подрядчик никогда не видит и не назначает пароль пользователя.

### DEC-030 — PIN и WebAuthn после первого входа

- **Статус:** утверждено.
- **Решение:** PIN разблокирует конкретное устройство; биометрия реализуется системным WebAuthn/passkey.

### DEC-031 — Кастомизация карточки объекта

- **Статус:** утверждено.
- **Решение:** аватар, фото обложки, цвет, градиент или контролируемый узор.
- **Отклонено:** произвольный CSS и свободный редактор макета.

### DEC-032 — Статический QR после базовой смены

- **Статус:** предварительно утверждено.
- **Решение:** первый рабочий сценарий может использовать кнопку; статический QR добавляется после проверки смены.
- **Отложено:** динамический QR.

### DEC-033 — Нет постоянной геолокации

- **Статус:** утверждено.
- **Решение:** координаты возможны только в значимые моменты и создают предупреждение, а не автоматическое обвинение.

### DEC-034 — Отчёты не ранжируют работников автоматически

- **Статус:** утверждено.
- **Решение:** индивидуальные и командные результаты разделены; сравнение допустимо только для сопоставимых работ.

### DEC-035 — Один frontend для трёх ролей

- **Статус:** предварительно утверждено как архитектурное решение.
- **Решение:** React/TypeScript PWA с разными оболочками и правами.

### DEC-036 — Модульный монолит

- **Статус:** предварительно утверждено как архитектурное решение.
- **Решение:** NestJS, PostgreSQL, REST, WebSocket, IndexedDB/Dexie, S3-совместимое хранилище.
- **Отклонено:** преждевременные микросервисы.

### DEC-037 — Мультитенантность с первого дня

- **Статус:** утверждено.
- **Решение:** каждая бизнес-запись принадлежит организации; права и изоляция проверяются сервером.

### DEC-038 — Финансы, склад и техника после пилота

- **Статус:** утверждено.
- **Решение:** эти модули не входят в MVP и не должны тормозить проверку ядра продукта.

### DEC-039 — Нативное приложение после подтверждения ценности

- **Статус:** утверждено.
- **Решение:** backend и sync проектируются API-first, но сначала запускается PWA.

### DEC-040 — Последовательная разработка по стадиям

- **Статус:** утверждено.
- **Решение:** одновременно активны одна основная стадия и один короткий production-срез; `docs/NOW.md` фиксирует его границы и условия завершения.

### DEC-041 — Прототип зафиксирован, production идёт вертикальными срезами

- **Статус:** утверждено 19 июля 2026.
- **Решение:** текущие макеты всех трёх ролей сохраняются как UI/UX-эталон. Production web и API развиваются отдельно в `apps/` и `packages/`.
- **Порядок:** каждый срез соединяет данные, backend-права и затронутые ролевые интерфейсы; оставшиеся UX-разрывы закрываются внутри соответствующего среза.
- **Следствие:** формальная незавершённость прежнего рубежа G2 больше не блокирует инженерный фундамент.
