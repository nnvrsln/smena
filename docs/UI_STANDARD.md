# Единый UI-стандарт СМЕНЫ

Этот документ обязателен для production-интерфейсов в `apps/web`. Его задача — не дать разделам продукта разойтись по композиции, типографике и языку.

## Проверка перед разработкой

До изменения экрана разработчик сверяет:

1. `NOW.md` — текущий этап и границы работы.
2. Профильный продуктовый документ — бизнес-правила выбранного раздела.
3. `SCREEN_MAP.md` — состав экрана и responsive-состояния.
4. `ACCEPTANCE_CRITERIA.md` — проверяемое поведение.

Если код, задача и документы расходятся, расхождение сначала фиксируется и устраняется. Новые поля нельзя добавлять только ради заполнения макета.

## Композиция раздела подрядчика

Каждый основной раздел использует одну последовательность блоков:

1. Hero: контекст раздела, один заголовок H1, одно короткое описание и только реально работающий основной action.
2. Сводка: до трёх ключевых показателей на одинаковых карточках.
3. Панель списка: название набора, количество элементов и необходимые фильтры.
4. Основной контент: карточки или таблица в зависимости от сценария из `SCREEN_MAP.md`.
5. Явные состояния загрузки, ошибки и пустого результата.

Hero и сводка не изобретаются заново для каждого раздела. В коде для них используются общие классы `contractor-page__*` и токены `--workspace-*`.

Глобальный header кабинета имеет одинаковый состав во всех разделах. На desktop он содержит поиск, текущий период и уведомления; на mobile — логотип и уведомления. Контекстные действия вроде создания объекта в глобальный header не добавляются.

## Визуальные правила

- Максимальная ширина, внутренние поля и фон задаются оболочкой кабинета.
- Вертикальный шаг раздела: 20 px на desktop, 14 px на mobile.
- Hero: минимальная высота 174 px, внутренние поля 28 × 30 px, радиус 24 px.
- Основные карточки: радиус 20 px, единая рамка, тень и hover-состояние.
- Карточка сотрудника использует чистую поверхность без декоративной обложки: самостоятельный аватар, имя, роль, контакт, фактические назначения и один главный action внизу.
- Статус присутствия сотрудника рассчитывается только из серверных событий текущего дня: открытая смена, завершённая смена или отсутствие выхода сегодня. Декоративные статусы и выдуманные поля запрещены.
- Состав объекта управляется в боковой панели из карточки объекта: один список сотрудников, явное выбранное состояние, фактический статус смены и одно атомарное сохранение.
- Сетка карточек: 3 колонки на desktop, 2 на tablet, 1 на mobile.
- Заголовок страницы: `clamp(27px, 2.5vw, 38px)`; заголовок карточки: 18 px.
- Текст интерфейса не может быть меньше 12 px.
- Цвет не используется как единственный способ передать статус.
- Все интерактивные элементы имеют видимый `focus-visible`; горизонтального скролла страницы быть не должно.

## Язык продукта

Интерфейс описывает работу пользователя, а не внутреннее устройство системы.

Нельзя показывать:

- названия баз данных, API и серверных технологий;
- внутренние номера этапов, спринтов и задач;
- обещания безопасности без пользовательского действия;
- пояснения, уместные только в прототипе;
- неработающие кнопки и будущие возможности.

Показатели, которые можно рассчитать из фактических данных, не вводятся вручную. Например, численность назначенной команды объекта считается из назначений сотрудников.

## Проверка готовности

Перед передачей изменения проверяются:

- соответствие четырём документам из раздела «Проверка перед разработкой»;
- desktop 1440 px, tablet 1024 px и mobile 375 px;
- отсутствие обрезанного текста и горизонтального скролла;
- клавиатурный фокус, loading, empty и error состояния;
- typecheck, автоматические тесты и production-сборка;
- отсутствие ошибок в консоли браузера.
