# СМЕНА — что делаем сейчас

Это короткий источник истины для ежедневной разработки. Детальные продуктовые правила остаются в остальных документах, но текущая работа определяется здесь.

## Активный этап

`03 — Инженерный фундамент`

## Активный срез

`F01 — Identity, роли и границы данных`

Цель: создать production-каркас, в котором один пользователь получает только свою организацию, доступные объекты, навигацию и разрешённые действия.

## Сейчас делаем

- сохраняем текущий прототип как UI/UX-эталон;
- развиваем production-код только в `apps/` и `packages/`;
- создаём один web-клиент для трёх ролевых оболочек;
- создаём модульный API;
- используем PostgreSQL как источник identity, организаций и объектов;
- проверяем матрицу разрешений автоматическими тестами.

## Сейчас не делаем

- не добавляем новые экраны в старый прототип;
- не переносим весь макет одним большим изменением;
- не начинаем задачи, табель, offline и уведомления до завершения F01;
- не дублируем доменные правила отдельно для каждой роли;
- не считаем development-identity настоящей авторизацией.

## Граница между кодом

| Путь | Назначение | Можно развивать |
|---|---|---|
| `src/`, `public/`, корневой `vite.config.js` | Замороженный визуальный прототип | Только критические исправления эталона |
| `apps/web` | Production PWA | Да |
| `apps/api` | Production API | Да |
| `packages/contracts` | Общие типы API и разрешений | Да |
| `infra/postgres` | Схема и миграции PostgreSQL | Да |

## Условия завершения F01

- production web и API запускаются одной командой;
- API возвращает контекст подрядчика, бригадира и рабочего;
- каждая роль получает только разрешённые объекты;
- навигация и действия строятся из разрешений;
- backend повторно проверяет полномочия;
- тесты матрицы ролей и изоляции проходят;
- PostgreSQL-миграция готова к запуску;
- отсутствующие локальные зависимости описаны честно.

## Прогресс F01 на 19 июля 2026

Готово:

- созданы `apps/web`, `apps/api` и `packages/contracts`;
- инициализирован Git-репозиторий и исключены временные артефакты;
- NestJS API и production web запускаются вместе;
- контекст трёх ролей приходит через API;
- подрядчик получает 3 объекта, бригадир 2, рабочий 1;
- navigation и permissions определяются общим контрактом;
- проходят 7 unit/integration-тестов, включая tenant isolation на реальной базе;
- PostgreSQL 18.4 работает локально внутри проекта, схема и dev-данные применены;
- `PostgresAccessRepository` возвращает 3/2/1 объекта для трёх ролей и не выдаёт объект другой организации;
- прототип продолжает собираться отдельно.

Осталось до завершения F01:

- добавить серверную сессию вместо development-заголовка роли;
- убрать development-репозиторий из runtime после подключения сессии.

## Следующее конкретное действие

Добавить серверную сессию и определять текущего пользователя по ней, а не по development-заголовку роли. До этого задачи, смены и дополнительные экраны не начинаются.
