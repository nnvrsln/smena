# СМЕНА — что делаем сейчас

Это короткий источник истины для ежедневной разработки. Детальные продуктовые правила остаются в остальных документах, но текущая работа определяется здесь.

## Активный этап

`03 — Инженерный фундамент`

## Активный срез

`S03 — Эксплуатационный фундамент`

Цель среза — сделать текущий production-код воспроизводимо проверяемым вне локальной машины и закрыть оставшиеся условия G3.

## Что входит в S03

- разделяем и документируем development, test, staging и production-конфигурации;
- делаем применение PostgreSQL-миграций воспроизводимой командой;
- добавляем CI-проверку типов, API-тестов и production-сборки;
- вводим безопасный health-check, единый формат ошибок и структурированные серверные логи;
- добавляем проверяемые команды резервного копирования и восстановления PostgreSQL;
- сохраняем автоматическую проверку изоляции организаций;
- документируем запуск, диагностику и восстановление staging-контура.

## Что не входит в S03

- не добавляем новые продуктовые экраны и доменные сущности;
- не начинаем задачи, offline, push, уведомления и месячное закрытие;
- не выбираем облачного провайдера и не публикуем production без отдельного решения;
- не переносим секреты в репозиторий;
- не изменяем замороженный визуальный прототип.

## Граница между кодом

| Путь | Назначение | Можно развивать |
|---|---|---|
| `src/`, `public/`, корневой `vite.config.js` | Замороженный визуальный прототип | Только критические исправления эталона |
| `apps/web` | Production PWA | Да |
| `apps/api` | Production API | Да |
| `packages/contracts` | Общие типы API и разрешений | Да |
| `infra/postgres` | Схема и миграции PostgreSQL | Да |

## F01 завершён

- production web и API запускаются одной командой;
- API возвращает контекст подрядчика, бригадира и рабочего;
- каждая роль получает только разрешённые объекты;
- навигация и действия строятся из разрешений;
- backend повторно проверяет полномочия;
- тесты матрицы ролей и изоляции проходят;
- PostgreSQL-миграции применены;
- вход по телефону и паролю создаёт серверную сессию в HttpOnly cookie;
- роль определяется сервером, development-заголовок удалён из runtime;
- выход отзывает сессию в PostgreSQL.

Готово:

- созданы `apps/web`, `apps/api` и `packages/contracts`;
- инициализирован Git-репозиторий и исключены временные артефакты;
- NestJS API и production web запускаются вместе;
- контекст трёх ролей приходит через API;
- подрядчик получает 3 объекта, бригадир 2, рабочий 1;
- navigation и permissions определяются общим контрактом;
- проходят 8 unit/integration-тестов, включая tenant isolation на реальной базе;
- PostgreSQL 18.4 работает локально внутри проекта, схема и dev-данные применены;
- `PostgresAccessRepository` возвращает 3/2/1 объекта для трёх ролей и не выдаёт объект другой организации;
- создан первый production-экран входа и защищённые кабинеты трёх ролей;
- production-кабинет подрядчика перенесён из утверждённого прототипа с той же desktop/mobile-композицией, KPI, объектными карточками, блоками внимания и план/факт;
- подрядчик получает из PostgreSQL управляемый список сотрудников и меняет их назначения на объекты в desktop/mobile-интерфейсе;
- backend запрещает рабочему управление сотрудниками и отклоняет назначение объекта другой организации;
- сервер хранит только хеш пароля и хеш токена сессии;
- desktop-вход, кабинеты подрядчика/бригадира и mobile-кабинет рабочего проверены живым браузерным проходом без console errors;
- рабочий начинает смену свайпом после обязательного подтверждения статического QR-кода назначенного объекта;
- открытая смена и событие её начала сохраняются в PostgreSQL, повторный старт и QR чужого объекта отклоняются;
- явка бригадира и KPI подрядчика считаются по открытым сменам и обновляются в web-клиенте каждые 5 секунд;
- API-тесты выполняются в отдельной PostgreSQL-схеме `smena_test`, не зависят от изменяемых dev-назначений и проходят `10/10`;
- рабочий завершает смену через итоговое окно, сервер закрывает открытую смену и сохраняет событие `shift_ended`;
- endpoint «сегодня» восстанавливает завершённый день после повторного входа, показывает приход, уход и фактическую длительность;
- после завершения смены явка бригадира и KPI подрядчика возвращаются к фактическому значению без ручного обновления;
- подрядчик и бригадир получают из PostgreSQL дневной табель только по разрешённым объектам: сотрудник, объект, приход, уход, длительность и статус;
- из строки табеля открывается серверная детализация дня с первичными событиями начала и завершения смены;
- подрядчик управляет объектами в отдельном адаптивном разделе: создаёт и редактирует название и обозначение, а численность назначенной команды рассчитывается автоматически;
- разделы «Объекты» и «Команда» используют единый production UI-стандарт, общую композицию, типографику и responsive-сетку;
- состав команды управляется с обеих сторон: из карточки сотрудника и из карточки объекта; сохранение состава объекта выполняется одной PostgreSQL-транзакцией;
- карточки команды и панель состава объекта показывают рассчитанные сервером состояния «На смене», «Смена завершена» и «Сегодня не выходил»;
- показатели «назначено» и «на смене» в карточке объекта рассчитываются из назначений и открытых смен, а изменения состава синхронно отражаются в разделах «Объекты» и «Команда»;
- отдельная карточка сотрудника объединяет профиль, текущие назначения, серверный статус дня и историю смен за выбранный месяц; она открывается из «Команды» и через детализацию дня в «Табеле»;
- история сотрудника ограничивается объектами, доступными текущей роли, показывает агрегаты периода и раскрывает первичные события конкретной смены;
- API валидирует объект, не допускает повторного обозначения и не позволяет бригадиру или другой организации менять данные;
- прототип продолжает собираться отдельно.
