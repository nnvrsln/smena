# Offline-first и синхронизация

## 1. Цель

Работник должен продолжать основные действия на объекте при плохом интернете, видеть честный статус и не терять данные после закрытия PWA.

Главный принцип:

> Сначала локальное сохранение, затем серверная синхронизация.

## 2. Предварительное условие

Для полноценной офлайн-работы пользователь должен хотя бы один раз войти и загрузить свои данные онлайн.

Первый вход, получение нового приглашения и загрузка неизвестной задачи без сети невозможны.

## 3. Доступно офлайн

Работнику:

- открыть ранее загруженные объекты и задачи;
- начать и завершить смену;
- отметить ручной обед;
- добавить объём;
- завершить этап;
- выполнить простую задачу;
- добавить комментарий;
- создать проблему;
- добавить фото;
- завершить смену.

Бригадиру:

- открыть сохранённую область;
- отметить сотрудника вручную;
- добавить результат;
- создать проблему и комментарий.

## 4. Недоступно или не гарантируется офлайн

- получить новую задачу с другого устройства;
- увидеть свежий ответ;
- проверить параллельную смену на сервере;
- окончательно подтвердить конфликтный объём;
- первый вход;
- получить серверный push;
- построить актуальный глобальный отчёт.

## 5. Локальные хранилища

IndexedDB/Dexie хранит:

- профиль и права;
- активные объекты;
- задачи и этапы;
- смены;
- комментарии и проблемы;
- метаданные файлов;
- outbox;
- курсор сервера;
- статусы операций.

Cache Storage/Service Worker хранит оболочку приложения и безопасные статические ресурсы.

## 6. Формат локальной операции

Каждая операция содержит:

- UUID/idempotency key;
- пользователя;
- устройство;
- организацию;
- тип команды;
- сущность;
- payload;
- время устройства;
- базовую версию;
- зависимости;
- количество попыток;
- статус;
- последнюю ошибку.

Примеры команд:

- `shift.start`;
- `break.start`;
- `task.progress.add`;
- `stage.complete`;
- `issue.create`;
- `message.send`;
- `file.prepare`;
- `shift.finish`.

## 7. Локальная транзакция

После действия приложение в одной локальной транзакции:

1. Создаёт outbox-операцию.
2. Обновляет локальное представление.
3. Показывает результат пользователю.
4. Запускает отправку, если сеть доступна.

Нельзя сначала менять интерфейс, а затем пытаться записать outbox отдельным ненадёжным действием.

## 8. Пользовательские статусы

### Отправлено

> ✓ Отправлено

Сервер подтвердил операцию.

### Сохранено на телефоне

> ◷ Сохранено на телефоне

Операция находится в outbox.

### Требует проверки

> ! Не удалось применить изменение

Нужен конфликтный сценарий или решение руководителя.

Технические слова `pending`, `retry`, `409` пользователю не показываются.

## 9. Глобальный индикатор

Примеры:

> Нет сети · 4 действия сохранены

> Отправляем 4 действия…

> Всё синхронизировано

> 1 действие требует проверки

Индикатор открывает список операций с понятным контекстом.

## 10. Триггеры синхронизации

- сразу после локального действия;
- событие появления сети;
- открытие приложения;
- возвращение приложения в foreground;
- ручная кнопка;
- Background Sync, если поддерживается;
- push-событие, если service worker может безопасно инициировать обновление.

Фоновая отправка закрытой PWA является best effort, а не единственной гарантией.

## 11. Порядок

Операции с зависимостями отправляются последовательно.

Пример:

1. `shift.start`;
2. `break.start`;
3. `break.finish`;
4. `shift.finish`.

Если начало не принято, завершение не должно примениться раньше него.

Независимые фото и комментарии можно загружать параллельно с ограничением нагрузки.

## 12. Идемпотентность

Сервер хранит UUID уже обработанной команды.

Повторная отправка:

- не создаёт второй объём;
- не начинает вторую смену;
- не публикует второй комментарий;
- возвращает прежний результат обработки.

## 13. Pull-изменения

Клиент хранит курсор последней успешной синхронизации.

Запрос pull получает только:

- изменения после курсора;
- доступные пользователю сущности;
- удаления/архивирование;
- новые версии;
- уведомления.

После применения курсор обновляется локально атомарно.

## 14. Типы конфликтов

### Независимые добавления

Фото и комментарии объединяются без конфликта.

### Аддитивный объём

Добавка +20 м² может примениться к обновлённому плану, если задача остаётся открытой и команда не дублирует один факт.

### Замена поля

Два разных срока или исполнителя требуют сравнения версий. Сервер не выбирает молча.

### Закрытая сущность

Поздний результат закрытой задачи становится «Поздний отчёт» и требует решения.

### Две командные сдачи

Разные общие итоги не суммируются. Подрядчик выбирает подтверждённое значение.

### Параллельные смены

Обе исходные отметки сохраняются, день получает конфликт.

## 15. Пример объёма

Офлайн:

- рабочий добавил +20 м²;
- подрядчик онлайн изменил план со 150 до 170.

После sync:

- план — 170;
- факт увеличивается на 20;
- история содержит оба действия.

Если подрядчик уже закрыл задачу, +20 попадает на проверку.

## 16. Пример смены

Работник начал в 07:54 офлайн, событие пришло в 12:10.

Сохраняются:

- время устройства;
- время сервера;
- задержка;
- способ;
- устройство.

Смена рассчитывается от 07:54, но может требовать проверки при аномалии часов или другого активного события.

## 17. Фото

Фото:

1. Сжимается в разумных пределах.
2. Сохраняется локально.
3. Получает миниатюру и outbox-запись.
4. Загружается отдельно.
5. После подтверждения локальный тяжёлый blob можно очистить по политике.

Комментарий или результат не должен ждать фото.

## 18. Ограничение хранилища

Приложение:

- оценивает доступное место;
- запрашивает persistent storage, если возможно;
- предупреждает при нехватке;
- не кэширует весь архив;
- приоритетно отправляет критичные операции;
- очищает только уже подтверждённые временные файлы.

## 19. Объём локального кэша

Работнику:

- активная смена;
- активные и ближайшие задачи;
- последние комментарии;
- собственный недавний табель;
- необходимые документы.

Бригадиру:

- назначенные объекты;
- активные сотрудники;
- открытые задачи и проблемы;
- последние смены.

Подрядчику глобальный архив доступен с сервера; локально сохраняется активный срез.

## 20. Выход с несинхронизированными действиями

Если пользователь пытается выйти или очистить устройство при непустом outbox:

- показывается предупреждение;
- предлагается синхронизировать;
- нельзя молча удалить очередь;
- принудительный выход требует отдельного подтверждения;
- администратор видит потерянные неподтверждённые операции только если они успели попасть на сервер — локальные данные другого устройства недоступны.

## 21. Обновление приложения

Новая версия PWA должна:

- мигрировать IndexedDB;
- сохранять outbox;
- не активировать несовместимый service worker посреди критической операции;
- показывать мягкое предложение обновиться;
- поддерживать хотя бы предыдущую версию sync-протокола на время развёртывания.

## 22. Несколько вкладок

Локальная БД едина для origin. Нужна координация вкладок:

- один лидер sync;
- блокировка двойной отправки;
- реактивное обновление UI;
- уникальные операции всё равно защищены сервером.

## 23. Безопасность

- локально хранится минимум нужных данных;
- секреты не кладутся в обычные записи;
- при отзыве сессии новые sync-команды не принимаются;
- после следующего онлайн-контакта доступ закрывается и кэш очищается по безопасной политике;
- файлы не становятся публичными из-за кэширования.

## 24. Метрики

Система отслеживает:

- длину outbox;
- возраст старейшей операции;
- процент повторов;
- конфликты;
- ошибки загрузки файлов;
- время до синхронизации;
- долю операций, созданных офлайн.

## 25. Не входит в MVP

- полноценное peer-to-peer взаимодействие устройств;
- гарантированная работа закрытой PWA во всех браузерах;
- синхронизация всего архива;
- автоматическое разрешение противоречивых общих результатов;
- нативный background service.
