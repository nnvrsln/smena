# Карта экранов и состояний

Версия: 0.3  
Стадия: 02 — UX-сценарии и актуализация прототипа  
Статус: первый рабочий контур

## 1. Принцип интерфейсов

Продукт использует одну React-кодовую базу и три ролевые оболочки:

- **подрядчик** — полноценный адаптивный web-кабинет с отдельной композицией для desktop и mobile;
- **бригадир** — mobile-first PWA для оперативного управления объектом;
- **рабочий** — mobile-first PWA с минимальным количеством ежедневных действий.

Одинаковая бизнес-сущность может иметь разное представление для роли. Это не три независимых продукта: переходы, права, данные и статусы остаются общими.

## 3. Глобальные состояния после входа

Эти состояния проектируются для каждой оболочки:

- первичная загрузка и skeleton;
- пустое состояние;
- нет доступа;
- объект архивирован;
- пользователь заблокирован;
- стабильная сеть;
- действие сохранено на устройстве;
- синхронизация;
- конфликт, требующий внимания;
- ошибка сервера с безопасным повтором;
- обязательное обновление приложения;
- непустой outbox перед выходом;
- центр активности;
- глобальный поиск в доступной области.

## 4. Подрядчик — основная навигация

### Desktop

Постоянная боковая навигация:

1. Обзор.
2. Объекты.
3. Команда.
4. Задачи.
5. Табель.
6. Отчёты.
7. Настройки.

Верхняя панель:

- глобальный поиск;
- рабочая дата/период;
- центр активности;
- быстрое создание задачи;
- выбор организации при наличии нескольких.

### Mobile web

Нижняя навигация содержит четыре частых раздела:

1. Обзор.
2. Объекты.
3. Задачи.
4. Команда.

Табель, отчёты и настройки доступны из меню. Кнопка создания задачи и центр активности остаются в верхней панели.

## 8. Универсальная задача — последовательность экранов

Создание использует progressive disclosure:

1. Что нужно сделать.
2. Объект и зона.
3. Исполнители.
4. Результат:
   - простое «готово»;
   - количество и единица;
   - количество с этапами.
5. План на день без обязательного времени начала/окончания.
6. Ответственный и проверяющий, только если нужны.
7. Дополнительные параметры: фото, приоритет, инструкция.
8. Проверка и создание.

Поля простого поручения не должны выглядеть как пустая сложная CRM-форма.

## 9. Responsive-правила подрядчика

### От 1200 px

- постоянный sidebar;
- четыре KPI в строку;
- три карточки объектов;
- таблицы с полными колонками;
- detail в side panel без потери контекста.

### 768–1199 px

- компактный sidebar;
- два KPI в строку;
- панели переносятся;
- таблицы сохраняют главные колонки, детали открываются отдельно.

### До 767 px

- нижняя навигация;
- KPI 2 × 2;
- объекты — горизонтальные snap-карточки;
- таблицы превращаются в информационные строки/карточки;
- формы и детали открываются полноэкранно;
- основное действие остаётся доступно большим пальцем.

## 11. Следующие UX-инкременты

1. Комментарий, официальный ответ, описание решения и связанная задача внутри проблемы.
2. История и локальные материалы проблемы при плохой сети; ролевые переходы рабочего и подрядчика.
3. Подробный табель и drill-down дня.
4. Инвайт и onboarding.
5. Полные offline/conflict состояния.
