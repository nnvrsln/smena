# Критерии приёмки MVP

Версия: 1.0  
Статус: черновик тестовой основы, ожидает ревью

## 1. Как пользоваться документом

Каждый критерий должен быть проверен на staging и, где указано, на реальном мобильном устройстве. Критерий считается выполненным только при наличии воспроизводимого результата и тестового подтверждения.

Обозначения:

- **P0** — без выполнения пилот запрещён;
- **P1** — обязательно для полноценного MVP, допустимо исправление до расширения пилота;
- **P2** — желательно, не блокирует ограниченный запуск.

## 2. Приглашение и вход

- **AUTH-01 / P0:** подрядчик создаёт приглашение с ролью и объектом; ссылка открывает правильный onboarding.
- **AUTH-02 / P0:** одно приглашение нельзя успешно использовать дважды.
- **AUTH-03 / P0:** отозванная или истёкшая ссылка не создаёт пользователя и показывает понятную причину.
- **AUTH-04 / P0:** onboarding сохраняет ФИО, специализацию, телефон и пароль; аватар можно пропустить.
- **AUTH-05 / P0:** занятый нормализованный номер телефона не создаёт дубликат учётной записи.
- **AUTH-06 / P0:** пользователь входит по телефону и паролю и остаётся в безопасной серверной сессии.
- **AUTH-07 / P1:** PIN разблокирует локальное приложение, но после отзыва сессии требует полный вход.
- **AUTH-08 / P1:** подрядчик создаёт одноразовый сброс пароля; старый пароль после смены не работает.
- **AUTH-09 / P0:** заблокированный сотрудник не может выполнить новую серверную операцию.
- **AUTH-10 / P1:** пользователь видит свои устройства и может отозвать выбранную сессию.
- **AUTH-11 / P2:** на поддерживаемом устройстве passkey/WebAuthn выполняет быстрый вход без передачи биометрии приложению.

## 3. Организации, объекты и люди

- **ORG-01 / P0:** данные одной организации недоступны участнику другой по прямому UUID, поиску, sync и файлам.
- **OBJ-01 / P0:** подрядчик создаёт объект с названием, часовым поясом и графиком.
- **OBJ-02 / P1:** карточка объекта поддерживает аватар и каждый тип фона: фото, цвет, градиент, узор.
- **OBJ-03 / P1:** оформление остаётся читаемым на светлой и тёмной фотографии.
- **OBJ-04 / P0:** сотрудника можно назначить на объект без команды и без бригадира.
- **OBJ-05 / P1:** объект можно архивировать без потери исторического табеля и задач.
- **OBJ-06 / P1:** можно создать зону и привязать к ней задачу.

## 4. Права и роли

- **PERM-01 / P0:** подрядчик назначает задачу рабочему напрямую и видит обратную связь.
- **PERM-02 / P0:** отсутствие бригадира не блокирует приглашение, смену, задачи, проверку и уведомления.
- **PERM-03 / P0:** бригадир не видит объекты и людей вне назначенной области.
- **PERM-04 / P0:** рабочий не может изменить чужую смену, роль или подтверждённый результат.
- **PERM-05 / P0:** скрытая кнопка на клиенте не считается защитой; запрещённый прямой API-запрос возвращает отказ.
- **PERM-06 / P1:** участник групповой задачи видит необходимый контекст команды без раскрытия лишних персональных данных.

## 5. Смена

- **SHIFT-01 / P0:** рабочий начинает смену одним действием; отметка сразу видна локально.
- **SHIFT-02 / P0:** невозможно создать две параллельные открытые смены одного человека.
- **SHIFT-03 / P0:** завершение показывает начало, окончание, обед и рассчитанное рабочее время.
- **SHIFT-04 / P0:** режим ручного обеда корректно учитывает start/end перерыва.
- **SHIFT-05 / P0:** режим фиксированного обеда считает настроенный интервал без дополнительных действий и показывает правило пользователю.
- **SHIFT-06 / P0:** офлайн-начало/окончание сохраняет время устройства и после сети синхронизируется без дубля.
- **SHIFT-07 / P1:** смена через полночь относится к правильной рабочей дате по настройке.
- **SHIFT-08 / P1:** перевод на другой объект создаёт сегменты одной смены, а не параллельные смены.
- **SHIFT-09 / P0:** забытая отметка исправляется запросом и решением; исходное событие остаётся доступно.
- **SHIFT-10 / P0:** смена руководителя за сотрудника помечается источником и автором.
- **SHIFT-11 / P1:** аномалии (нет окончания, отрицательный/слишком длинный интервал, конфликт обеда) попадают в список проверки.

## 6. Простые задачи

- **TASK-01 / P0:** подрядчик или бригадир создаёт простую задачу и назначает её одному рабочему.
- **TASK-02 / P0:** рабочий видит задачу на день без запуска таймера и отмечает «Готово».
- **TASK-03 / P0:** для простой задачи проверка может быть выключена; тогда она завершается сразу.
- **TASK-04 / P1:** при включённой проверке результат попадает проверяющему и может быть принят или возвращён.
- **TASK-05 / P0:** история показывает автора, назначившего, исполнителя, изменения и результат.

## 7. Объёмные и составные задачи

- **VOL-01 / P0:** задача хранит общий объём, единицу и отдельный дневной план.
- **VOL-02 / P0:** один исполнитель автоматически становится ответственным.
- **VOL-03 / P0:** рабочий несколько раз добавляет прогресс в течение дня; итог суммируется без повторной отправки одних данных.
- **VOL-04 / P0:** система различает общий план, дневной план, заявленный факт и подтверждённый факт.
- **VOL-05 / P0:** проверяющий принимает весь результат, часть результата или возвращает на доработку с причиной.
- **VOL-06 / P0:** старая версия результата остаётся в истории после доработки.
- **VOL-07 / P0:** незавершённый остаток переносится на следующий день без обнуления уже подтверждённого объёма.
- **VOL-08 / P0:** в один день у рабочего одновременно отображаются объёмная и несколько простых задач без требования переключать активный таймер.
- **VOL-09 / P1:** составная задача поддерживает один уровень этапов; этапы второго уровня создать нельзя.
- **VOL-10 / P1:** этап может быть простым или количественным.

## 8. Групповые задачи и ответственность

- **TEAM-01 / P0:** общую задачу можно назначить нескольким людям без искусственного деления общего объёма.
- **TEAM-02 / P0:** выбранный ответственный отправляет общий итог; участие остальных видно в табеле как командное.
- **TEAM-03 / P0:** задачу можно создать без ответственного; команда видит «Взять ответственность».
- **TEAM-04 / P0:** при одновременной попытке взять ответственность успешен один атомарный запрос.
- **TEAM-05 / P1:** руководитель передаёт ответственность другому участнику, причина и история сохраняются.
- **TEAM-06 / P0:** при отсутствии ответственного задача помечается как риск и остаётся управляемой назначившим.

## 9. Проблемы, комментарии и файлы

- **ISSUE-01 / P0:** рабочий создаёт проблему из задачи с категорией, влиянием, текстом и необязательным фото.
- **ISSUE-02 / P0:** проблема автоматически содержит объект, задачу, автора и время.
- **ISSUE-03 / P0:** блокирующая проблема уведомляет правильного руководителя при наличии и отсутствии бригадира.
- **ISSUE-04 / P1:** проблему можно подтвердить, взять в работу, решить, закрыть и повторно открыть.
- **COMM-01 / P0:** подрядчик и рабочий обсуждают задачу напрямую.
- **COMM-02 / P0:** комментарий сам не изменяет поля или статус задачи.
- **COMM-03 / P1:** упоминание создаёт запись активности адресату.
- **COMM-04 / P1:** важная инструкция поддерживает явное «Ознакомился».
- **FILE-01 / P0:** интерфейс показывает одно действие «Добавить фото», не заставляя выбирать бизнес-категорию «камера/галерея».
- **FILE-02 / P0:** файл недоступен пользователю без права на связанную сущность.
- **FILE-03 / P0:** неудачная загрузка повторяется; текст/результат при допустимом сценарии не теряется из-за фото.
- **FILE-04 / P1:** фото имеет автора, время и контекст и отображается миниатюрой.

## 10. Офлайн и синхронизация

- **SYNC-01 / P0:** после первой загрузки рабочий открывает текущий день без сети.
- **SYNC-02 / P0:** начало/окончание смены, простой результат, прогресс, проблема и комментарий сохраняются в outbox.
- **SYNC-03 / P0:** пользователь видит состояние каждой несинхронизированной операции.
- **SYNC-04 / P0:** outbox переживает закрытие и повторный запуск приложения.
- **SYNC-05 / P0:** десятикратный повтор одной операции создаёт один серверный результат.
- **SYNC-06 / P0:** зависимые операции применяются в правильном порядке, например локальная проблема после локально созданного результата.
- **SYNC-07 / P0:** после восстановления сети состояние сервера и клиента сходится без ручной перезагрузки данных.
- **SYNC-08 / P0:** противоречивое изменение не затирается молча и получает статус «Требует внимания».
- **SYNC-09 / P1:** после долгого офлайна пользователь получает сводный результат синхронизации.
- **SYNC-10 / P0:** обновление PWA не удаляет непустой outbox.
- **SYNC-11 / P0:** выход с непереданными данными предупреждает о последствиях и предлагает синхронизировать.
- **SYNC-12 / P0:** сценарии проверены с реальными разрывами сети на Android и iPhone целевых версий пилота.

## 11. Уведомления

- **NOTIF-01 / P0:** центр активности работает даже при запрещённом push.
- **NOTIF-02 / P1:** разрешение push запрашивается после объяснения и действия пользователя.
- **NOTIF-03 / P0:** уведомление открывает конкретную задачу, проблему, смену или проверку.
- **NOTIF-04 / P0:** одинаковые события группируются и не создают спам.
- **NOTIF-05 / P0:** рабочий не получает push о собственном действии.
- **NOTIF-06 / P0:** без бригадира требующее решения событие маршрутизируется подрядчику.
- **NOTIF-07 / P1:** тихие часы и профиль уведомлений применяются корректно.
- **NOTIF-08 / P1:** отзыв устройства прекращает доставку на его подписку.
- **NOTIF-09 / P1:** экран блокировки может скрывать чувствительный текст.

## 12. Табель и отчёты

- **REPORT-01 / P0:** экран «Сегодня» показывает явку, смены, задачи, объёмы и проблемы с раскрытием первичных записей.
- **REPORT-02 / P0:** месячный отчёт сотрудника содержит каждый день, часы, обед, объекты, задачи и корректировки.
- **REPORT-03 / P0:** клик по дню открывает события смены и связанные результаты.
- **REPORT-04 / P0:** клик по задаче открывает автора, назначившего, исполнителей, изменения, результаты и проверки.
- **REPORT-05 / P0:** командный объём не выдаётся за личную выработку каждого участника.
- **REPORT-06 / P0:** план, заявленный и подтверждённый факт визуально и семантически разделены.
- **REPORT-07 / P0:** Excel-выгрузка совпадает с выбранными фильтрами и версией периода.
- **REPORT-08 / P0:** закрытый период не меняется молча; корректировка создаёт новую версию.
- **REPORT-09 / P1:** журнал корректировок содержит старое/новое значение, причину, автора и согласовавшего.
- **REPORT-10 / P1:** фотоотчёт фильтруется по объекту, задаче и дню.

## 13. UX, адаптивность и доступность

- **UX-01 / P0:** рабочий проходит обычный день с фиксированным обедом двумя основными отметками смены.
- **UX-02 / P0:** на главном экране рабочего нет обязательного управления таймерами производственных задач.
- **UX-03 / P0:** подрядчик выполняет основные операции с телефона и компьютера.
- **UX-04 / P0:** ключевые кнопки и состояния читаемы на целевых размерах экрана и при солнечном освещении в разумных пределах.
- **UX-05 / P1:** интерактивные элементы имеют достаточную область касания, видимый фокус и доступные названия.
- **UX-06 / P1:** ошибки описывают, что произошло и что делать, без технических кодов как единственного текста.
- **UX-07 / P0:** пользователь всегда видит, сохранено ли действие только на устройстве или уже на сервере.
- **UX-08 / P1:** основные сценарии доступны с клавиатуры в кабинете подрядчика.

## 14. Безопасность и эксплуатация

- **SEC-01 / P0:** пароли хранятся только надёжным адаптивным хэшем; токены приглашений и сбросов — хэшами.
- **SEC-02 / P0:** нет секретов и production-ключей в клиентском bundle или репозитории.
- **SEC-03 / P0:** значимые изменения задач, смен, прав и периодов попадают в аудит.
- **SEC-04 / P0:** проверены rate limit и защита входа от перебора.
- **SEC-05 / P0:** приватный файл нельзя получить по угадываемой постоянной ссылке.
- **OPS-01 / P0:** production имеет автоматическое резервное копирование.
- **OPS-02 / P0:** до пилота проведено успешное тестовое восстановление.
- **OPS-03 / P0:** ошибки API, worker, push, файлов и sync наблюдаемы.
- **OPS-04 / P0:** существует инструкция реакции на инциденты и контакт поддержки.
- **OPS-05 / P1:** миграция откатывается либо имеет безопасный forward-fix план.

## 15. Производительность

Конкретные цифры уточняются после замеров, но пилот блокируется, если:

- главный экран на типичном телефоне и слабой сети не показывает локальные данные сразу;
- обычное локальное действие заметно зависает из-за ожидания сервера;
- список из 15–50 работников или задач теряет управляемость;
- отчёт за месяц регулярно завершается ошибкой;
- фото блокирует отправку текстового результата;
- sync после короткого разрыва требует ручной очистки данных.

## 16. Gate готовности к пилоту

Перед включением реальных сотрудников должны быть выполнены:

- все P0;
- нет открытых дефектов уровня потери данных, нарушения tenant isolation или неверного табеля;
- P1-дефекты имеют осознанное решение и владельца;
- пройден сквозной тест полного рабочего дня;
- пройден сквозной тест месяца на тестовых данных;
- пройдены offline/reconnect сценарии;
- настроены мониторинг, резервные копии и поддержка;
- подрядчик провёл самостоятельный пробный сценарий без подсказок разработчика.
